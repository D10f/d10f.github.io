<div class="sectionbody">
<div class="paragraph">
<p>I like the idea of a "minimal" Linux desktop because that means there are fewer things to understand, and the more effective I can be with it. However, I don&#8217;t like being dogmatic about things. Minimalism can become a very deep rabbit hole, and at the end of the day I need to be productive with my system. With that in mind, I&#8217;ll be making certain trade-offs in the name of convenience as I see fit for my personal use case.</p>
</div>
<div class="paragraph">
<p>The goal of this exercise is to explore the different components that make up a Linux desktop and integrate them separately.</p>
</div>
</div>

<div class="sect1"><h2 id="_graphical_environment">Graphical Environment<a class="anchor" href="#_graphical_environment">
                    <svg aria-hidden="true">

                    </svg>

<div class="sectionbody">
<div class="paragraph">
<p>After a bare installation of Debian we&#8217;ll be greeted by a login prompt at the console; no graphical environment yet. This is where you&#8217;ll need to log in as the root user, instead of your own non-privileged account, to install a few packages.</p>

<div class="sect2"><h3 id="_dmenu">dmenu<a class="anchor" href="#_dmenu">

<div class="paragraph">
<p>The Dynamic Menu, another simple utility that serves as an application launcher.</p>
</div>
<div class="paragraph">
<p>It reads newline-separated items from standard input, and returns whatever the user has selected to standard output. This can be leveraged in custom shell scripts to have some interactivity, as well, which is really useful.</p>

<div class="paragraph">

<div class="dlist">

</dd>
<dt class="hdlist1">dmenu-border-20230512-0fe460d </dt>

</dd>
<dt class="hdlist1">dmenu-fuzzymatch-5.3.diff </dt>

<div class="sect1"><h2 id="_login_startup_scripts">Login Startup Scripts<a class="anchor" href="#_login_startup_scripts">
                    <svg aria-hidden="true">
                        <use href="/sprite.svg#icon-link" />
                    </svg>
                </a></h2>
<div class="sectionbody">
<div class="paragraph">
<p>When I log in to my account, I&#8217;m used to seeing desktop icons, a background image, a taskbar loading, etc. But with this setup we only have a console environment that can&#8217;t even launch graphical applications such us Firefox, GIMP or even a window manager like dwm; we&#8217;ll see an error saying "cannot open display".</p>
</div>
<div class="paragraph">
<p>This is because we need to have an X display server up and running in order to launch graphical applications. On a typical installation there&#8217;s another piece of software we haven&#8217;t looked up yet, the "display manager", that handles all of this when we log in.</p>

            </div>

<div class="sect1"><h2 id="_privilege_escalation">Privilege Escalation<a class="anchor" href="#_privilege_escalation">
                    <svg aria-hidden="true">
                        <use href="/sprite.svg#icon-link" />
                    </svg>
                </a></h2>
<div class="sectionbody">
<div class="paragraph">
<p>Switching to the root user account to run certain commands that require elevated privileges is the best way to go in terms of security. But it&#8217;s a bit inconvenient for one-off tasks, and it would be nice to have that ability for at least a few selected commands.</p>
</div>
<div class="paragraph">
<p>This is what utilities like <code>sudo</code> are for, but perhaps you&#8217;ve noticed that this very common package is not already installed. We could do that, but in the pursuit of a minimalist system I think it&#8217;s worth exploring a few alternatives.</p>
</div>
<div class="sect2"><h3 id="_sudo_rs">sudo-rs<a class="anchor" href="#_sudo_rs">
                    <svg aria-hidden="true">
                        <use href="/sprite.svg#icon-link" />
                    </svg>
                </a></h3>
<div class="paragraph">
<p>This one is not just yet another re-write of classic GNU/Linux utilities in the Rust programming languages. It&#8217;s developed in collaboration with long time <code>sudo</code> maintainer Todd C. Miller, and it addresses several shortcomings of the original tool.</p>
</div>
<div class="paragraph">
<p>However, it&#8217;s still under heavy development and has yet to see "real world usage" to polish all the rough edges. It also includes a lot of features that the average user probably needs (speaking strictly from a desktop user perspective).</p>
</div>
<div class="paragraph">
<p>Because of that, I prefer going with a more minimalistic approach.</p>
</div>
</div>
<div class="sect2"><h3 id="_opendoas">OpenDoas<a class="anchor" href="#_opendoas">
                    <svg aria-hidden="true">
                        <use href="/sprite.svg#icon-link" />
                    </svg>
                </a></h3>
<div class="paragraph">
<p>FreeBSD&#8217;s <code>doas</code> ported to other UNIX systems. It&#8217;s a much smaller code base, partly because it also offers fewer features in comparison to <code>sudo</code>, which helps reducing the attack surface. However, the main repository highlights that this is not a security guarantee:</p>
</div>
<div class="quoteblock">
<blockquote>
There are fewer eyes on random doas ports, just because sudo had a vulnerability does not mean random doas ports are more secure if they are not reviewed or PAM is configured incorrectly.
</blockquote>

&#8212; Duncan Overbruck<br>
