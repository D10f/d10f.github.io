<div class="sectionbody">
<div class="paragraph">
<p>When deploying services using Docker Swarm, it&#8217;s up to the orchestrator engine to decide where containers will be deployed. On a single node it&#8217;s easy to predict in advanced the host, and we can work with some assumptions. However, on more than one node this becomes a little more difficult.</p>
</div>
</div>
</div><menu class="toc"><li class="toc__item"><a href="#_problem_statement">Problem Statement</a></li><li class="toc__item"><a href="#_initial_setup">Initial Setup</a></li><li class="toc__item"><a href="#_docker_service_templates">Docker Service Templates</a></li></menu>
<div class="sect1"><h2 id="_problem_statement">Problem Statement<a class="anchor" href="#_problem_statement">
                    <svg aria-hidden="true">
                        <use href="/sprite.svg#icon-link" />
                    </svg>
                </a></h2>
<div class="sectionbody">
<div class="paragraph">
<p>You have set up a fleet of hosts running Docker Swarm, on top of which you can now run various containerized applications. To monitor the health of the hosts you want to run metric collection software such as Telegraf, InfluxDB and Grafana.</p>
</div>
<div class="paragraph">
<p>You take advantage of the fact that your fleet already spawns multiple hosts and has an orchestrator taking over the deployment of individual containers, and you write compose file to make things easier for yourself.<br>
Unfortunately, when you check the dashboard you see that while the data points are correctly being reported, the hostnames correspond to the individual Docker container IDs, instead of the actual hosts that you care to monitor. This makes it difficult to distinguish which hosts are under more or less load.</p>
</div>
</div>
</div>

<p>However, the way it works is by invoking Go&#8217;s <code>os.Hostname</code> function to populate the field, which reads directly from <code>/proc/sys/kernel/hostname</code>. It would be reasonable to assume that, after mounting the entire <code>/hostfs/proc</code> to the container, this information would be available. Unfortunately this is not the case and the hostname that is reported is the ID of the Docker container in which it runs.</p>
