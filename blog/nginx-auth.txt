<div class="sect1"><h2 id="_password_protected_routes">Password-protected Routes<a class="anchor" href="#_password_protected_routes">
                    <svg aria-hidden="true">
                        <use href="/sprite.svg#icon-link" />
                    </svg>
                </a></h2>
<div class="sectionbody">
<div class="paragraph">
<p>Nginx supports the <a href="https://www.rfc-editor.org/rfc/rfc7617.html">HTTP Basic Authentication scheme</a> through the use of the <code>auth_basic</code> and <code>auth_basic_user_file</code> directives. This authentication scheme is quite straight forward: a simple challenge-response, where the user sends the username and password that the client then validates against a password file stores on the server.                 <p>RFC 7617, as well as the original RFC 2617, refer to the server implementing the Basic Access Authentication scheme as the "client", and visitors who need to pass the challenge-response as the "users". This post follows this terminology.</p>

                <p>This authentication scheme sends the challenge-response from the client in clear text (base64-encoded). It SHOULD NOT be used without additional security enhancements such as HTTPS.</p>
            </div>
<div class="paragraph">
<p>Due to the sensitive nature of this file, passwords stored in this file are expected to be hashed using one of the following hashing algorithms: MD5, APR1 (a variant of MD5 used by Apache), SHA256, SHA512 or Bcrypt (otherwise authentication will fail). There are two well-known utilities that can be used to generate passwords in this format:</p>
</div>

<div class="paragraph">
<p>Instead of providing the password as an argument you can use the <code>-in</code> option to specify a file, or the <code>-stdin</code> to read from standard input. This can be helpful to decrease the risk of leakage due to passwords appearing in the shell&#8217;s history to in a process monitor.</p>
</div>
<div class="paragraph">
<p>Unlike <code>htpasswd</code>, it won&#8217;t create a new file for storing the username and password pair, nor it will append to it. It simply outputs the computed hash of the given password, and is up to you to update the file however you choose to. It may be more convenient to use <code>htpasswd</code> instead of <code>openssl</code> because of this, but it may also not be possible to install new software on certain environments, whereas OpenSSL is a popular library likely to be included in that case.</p>
